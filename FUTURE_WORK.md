## 1. 🛡️ Снижение хвостового риска стратегии (tail risk) — ПРИОРИТЕТ

**Дата постановки:** 2026-06-16. **Статус:** проанализировано, ждёт реализации + бэктеста.

### Проблема (два разных хвостовых риска)
- **Per-trade tail.** SL у Put = `+150%` (`sl_pct=1.50`): один стоп ≈ −150% кредита ≈ **1.8×
  среднего выигрыша**. «Толщина» одного убытка велика.
- **Cluster tail (главный).** Серия стопов в неблагоприятном режиме. У Pure Put были месяцы
  −50/−65/−80%; V2 hybrid сократил до 2 убыточных месяцев, но один всё равно **−42%** (Sep).
  CB ловит кластер только после 5 убытков; даже с dynsize ×0.5 за месяц набегает много.
  При $400 и 15% маржи/сделку кластер из 5 убытков всё равно заметно бьёт по счёту.

### Меню рычагов

| # | Рычаг | Лечит | Риск для edge | Сложность | Бэктест |
|---|---|---|---|---|---|
| 1 | **Дневной/недельный лимит убытка** (пауза после −X% реализованного за период) | cluster ⭐ | низкий | низкая | да (выбрать X) |
| 2 | **Лимит концентрации** (max N одновременных; max на сторону/страйк) | cluster ⭐ | низкий | низкая | желательно |
| 3 | **Жёстче CB** (5→3 убытка; дольше пауза) | cluster | средний (ложные паузы) | низкая | да |
| 4 | **Меньше сайзинг** (`MARGIN_PCT_PER_TRADE` 15%→~10%, или DD-scaled) | оба, линейно | нулевой | низкая | нет |
| 5 | **Put-спред** (продать ATM + купить OTM Put → defined risk) | per-trade ⭐⭐ структурно | меняет стратегию | **высокая** | да |
| 6 | ~~Тоньше SL (1.50→1.00)~~ | per-trade | **ОПАСНО** — short-vol с тесным стопом кровит на шуме, фиксирует убыток там, где theta вытащила бы. Только через бэктест. | низкая | обязательно |

### Рекомендованный путь (слоистый overlay)
- **Tier 1 (первым; бьёт в cluster tail, edge почти не трогает):** дневной лимит убытка
  + лимит концентрации. Прогнать сетку порогов на `variant_backtest.py` против плохих
  месяцев — критерий: режет ли месяц −42% и max-DD без потери годового edge (+7.09%/сделку).
- **Tier 2:** снизить/DD-масштабировать сайзинг. Линейно режет хвост, edge не трогает.
- **Tier 3:** Put-спред — структурный кап max-loss; большой проект (движок бэктеста не умеет
  спреды — нужна доработка `_simulate_short_premium` + ре-валидация + два лега на Bybit).

### ⚠️ Важное несоответствие для закрытия в Tier 1
`execution_config.py` **уже** содержит `LIVE_DAILY_LOSS_LIMIT_USDT=100`, но **paper-луп
(`paper_loop.py`) его не реализует.** То есть дневной лимит убытка задуман для live, но в
paper не работает → бэктест/paper и live разойдутся. Реализовать единый дневной лимит в
`paper_loop.loop()` (halt новых открытий после −X% реализованного за UTC-сутки).

## 5. 🐛 Критические баги (код-ревью 2026-06-16)

> **Ждёт:** доказательства прибыльности paper trade. Фиксить и улучшать код, пока
> стратегия не доказала edge, — пустая трата времени.

### 5.1 `paper_repo.py` — отравление пула соединений
Все 17+ функций: `get_conn() → try → commit → finally → put_conn()`. При ошибке
между `get_conn()` и `commit()` соединение возвращается в пул **в состоянии aborted
transaction** — psycopg2 отвергнет все последующие запросы на нём.
**Фикс:** добавить `except: conn.rollback(); raise`.

### 5.2 `paper_loop.py` — race condition на circuit breaker
`record_trade_result()` вызывает `get_state()` → модифицирует → `update_state()` в **двух
транзакциях**. Если две позиции закрываются в одной итерации — одно обновление теряется,
CB может не сработать при 5 подряд убытках.
**Фикс:** атомарный `UPDATE ... SET consec_losses = consec_losses + 1 RETURNING`.

### 5.3 `reconcile.py` — reconciled-позиции записывают PnL = 0
Позиции, закрытые вне бота (мануал, ликвидация, экспирация), получают `pnl_usd=0.0`.
Искажает статистику (`position_stats`) и **обходит daily loss limit** — убыток от ликвидации
не учитывается.
**Фикс:** рассчитывать PnL на основе `entry_price` + mark/intrinsic.

### 5.4 `broker.py` — нет exception handling вокруг вызовов биржи
`sell_to_open()` / `buy_to_close()` могут бросить сетевые/API исключения. Необработанное
исключение mid-retry оставляет систему в неконсистентном состоянии.
**Фикс:** try/except с логом, пометка позиции для reconcile.

### 5.5 `paper_loop.py` — нет лимита на количество позиций
Есть margin cap 80%, но нет жёсткого `MAX_OPEN_POSITIONS`. В волатильном рынке может
открыть десятки позиций. Пересекается с п.2 (лимит концентрации).
**Фикс:** `MAX_OPEN_POSITIONS` в `execution_config.py` (рекомендация: 5–10).

### 5.6 `paper_loop.py` — деление на ноль `entry_credit_pct`
Строка 341: `entry_credit_pct = ... / spot * 100` — paper-ветка не проверяет `spot > 0`
(live-ветка проверяет). При `spot == 0` → `ZeroDivisionError`.

### 5.7 `paper_repo.py` — `open_position` возвращает `0` при ошибке
`return int(row[0]) if row else 0` — `0` выглядит как валидный ID. Caller может работать
с несуществующей позицией. **Фикс:** возвращать `None`.

## 6. ⚠️ Предупреждения и улучшения (код-ревью 2026-06-16)

> **Ждёт:** аналогично п.5.

| # | Файл | Проблема |

| 6.7 | `reconcile.py` | Telegram спам при повторяющемся расхождении (нет дедупликации) |
| 6.8 | `paper_repo.py` | `close_position` unknown exit_reason → тихий fallback `"closed_time"` |
| 6.9 | `paper_loop.py` | Stale `open_pos_now` для margin check (позиции закрытые ранее в итерации) |

### Общие проблемы кодовой базы

| Проблема | Влияние |
|----------|---------|
| `print()` вместо `logging` — все модули | Нет уровней логов, ротации, structured logging |
| `float` для денег повсюду | Накопление ошибок округления |
| `SimpleConnectionPool` не thread-safe | Коррупция при многопоточности |
| Нет transaction isolation в `paper_repo` | Невозможны атомарные мульти-шаговые операции |
| Hardcoded credentials в `docker-compose.yml` | `user` / `password` — нужен `.env` |
| Нет healthcheck в docker-compose | Race при старте сервисов |

## 8. 📈 Улучшение профита: Динамический Sizing по ADX Score

**Статус:** проанализировано, прогнан sweep-бэктест на 365 днях (2026-06-16). Ждёт реализации.

### Описание
Текущая стратегия V2 Hybrid использует ADX/regime как бинарный фильтр (range/transition = торгуем, trend = не торгуем).
Разработан градиентный **ADX Readiness Score (0-10)** (учитывает значение ADX, угол наклона за 6ч и спред +DI/-DI).

По результатам бэктестов абсолютным победителем стал подход **"Baseline + Sizing"**:
1. Полностью сохраняем текущий бинарный фильтр (сохраняет 582 сделки/год, плавную кривую и всего 2 убыточных месяца).
2. Используем ADX Score для изменения размера позиции:
   - `Score 0-7`: обычный объем (1.0x).
   - `Score 8-10` (идеальные условия): увеличенный объем (1.5x).

**Результаты бэктеста:**
- **Без сайзинга (текущая):** Прибыль +$5,547, Средняя на сделку +$9.53.
- **С сайзингом:** Прибыль **+$7,181**, Средняя на сделку **+$12.34**.

### Что нужно сделать
Внедрить логику множителя в `live_sizing.py`:
- `adx_score` уже рассчитывается и прикрепляется к словарю `extra` сигнала (в `strategy_registry.py`).
- В функции `realistic_size_lots` (или аналоге) проверять `signal.get('adx_score', 0)`.
- Если `score >= 8`, применять множитель `1.5` к итоговому размеру позиции (не превышая глобальные лимиты).
