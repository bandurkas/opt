## 0. 🎯 Приоритеты к live-валидации стратегии (2026-06-16)

> Ранжировано по одному критерию: **что быстрее всего и надёжнее всего приближает нас
> к доказательству edge в бою** — т.е. к прохождению go-live гейта (`PROJECT_DOSSIER.md` §8.3)
> на ДОСТОВЕРНЫХ данных. Гейт: (1) ≥20–30 циклов в разных режимах, (2) paper avg в пределах
> 30–50% бэктеста, (3) SL+CB+dynsize наблюдались вживую, (4) концентрация под контролем.

### ⛔ Реальный блокер №1 (не пункт бэклога, но определяет всё)
Гейт стоит на **8/≥20–30 циклов**, потому что рынок в `regime=trend` → Put дисквалифицирован
→ бот корректно простаивает, новые циклы не копятся. Это не баг. Варианты ускорить набор
выборки (любой — ТОЛЬКО через бэктест, чтобы не сломать edge):
- принять как есть, ждать смены режима (самый честный путь, нулевой риск для edge);
- проверить, не слишком ли узок range-детектор / Call-сторона (см. `regime.py`) — возможно,
  валидные range-окна отсекаются. Бэктест на покрытие сделок, а не только на edge.
- НЕ форсировать сделки в тренде ради счётчика — это отравит саму выборку, которую валидируем.

### P0 — Достоверность paper-данных (без этого валидация бессмысленна)
Пока эти баги живы, статистика g-гейта может молча искажаться или бот может «замолчать»
(как уже было с SQL-багом — 3 дня тишины, 0 закрытий). Чинить ПЕРВЫМИ.
- **§5.2 race на circuit breaker** — гейт п.3 ТРЕБУЕТ наблюдать CB вживую; потеря апдейта при
  двух закрытиях в одной итерации → CB может не сработать на 5 убытках → не увидим защиту
  (или увидим ложно) И пропустим кластерный риск. ⭐ самый важный для доверия к гейту.
- **§5.1 отравление пула соединений** — aborted-транзакция в пуле → тихие сбои записи позиций
  и `position_stats` → данные гейта портятся незаметно.
- **§5.6 `ZeroDivisionError` при `spot==0`** — может уронить paper-loop (повтор «молчания»).
- **§5.7 `open_position` возвращает `0`** — ложно-валидный ID → работа с несуществующей
  позицией → порча данных.

### P1 — Корректность исполнения для перехода в live (нужно ДО реальных денег, после гейта)
- **§2 / §5.5 лимит концентрации** — ✅ **СДЕЛАНО 2026-06-16** (`MAX_OPEN_POSITIONS=4`, коммит

## 1. 🛡️ Снижение хвостового риска стратегии (tail risk) — ПРИОРИТЕТ

**Дата постановки:** 2026-06-16. **Статус:** ✅ **ВНЕДРЕНО 2026-06-16** (коммит `34e150b`):
концентрационный кэп `MAX_OPEN_POSITIONS=4` (победитель бэктеста) реализован в paper+live.
Дневной лимит убытка — ОТВЕРГНУТ бэктестом, НЕ внедряется. Исходное исследование ниже.

### ✅ Вывод бэктеста (harness `backend/services/tail_overlay_sweep.py`, коммит `987efab`)
Event-driven портфельный replay (open/close, без look-ahead) на production-сигналах (v3),
сплит train/holdout (последние 90д). Все цифры подтверждены **out-of-sample**:

| Рычаг | Вердикт | full edge / худший мес / убыт.мес | holdout edge / maxDD |
|---|---|---|---|
| **`MAX_OPEN_POSITIONS=4`** | ✅ **ПОБЕДИТЕЛЬ** | **+7.7%** / **−11.3** / 2 | **+11.4%** / 29% |
| (без лимита, как сейчас) | базовый | +4.9% / −33.7 / 2 | +7.5% / 91% |
| `max_open=3` | альтернатива | +7.8% / −11.1 / 3 | +11.8% / 24% |
| Дневной лимит убытка | ❌ **ОТВЕРГНУТ** | хуже edge **и** хуже худший мес (−18.2) | — |
| Жёстче/любой CB | ❌ **ОТВЕРГНУТ** | режет edge | +7.5→−2.7% |
| По-сторонний лимит | ❌ хуже глобального | ниже edge, больше убыт.мес | — |
| `dyn_size` (уже в проде) | ✅ оставить | снижает DD «бесплатно» | — |

**Главное:** жёсткий лимит одновременных позиций (≈4) **одновременно** режет худший месяц
(−33.7%→−11.3%) **и усиливает** edge (+4.9→+7.7%, holdout +11.4%) — отсекаются именно
кластерные сделки с отрицательным EV. Это чистый риск-оверлей, **не трогает entry/exit**.

**Сюрприз для прода:** даже `max_open=8` (≈ текущий 80%-маржинальный кап) на полном годе
взрывается — у живого бота **нет жёсткого лимита числа позиций**, только мягкий маржинальный.
Кандидат на отключение: live-CB 5/48h (в бэктесте скорее вредит).

**К внедрению (когда решим):** `MAX_OPEN_POSITIONS=4` в `execution_config.py` + гейт в
`paper_loop.loop()` перед открытием (инертно к edge). Дневной лимит убытка из плана НИЖЕ —
отвергнут данными, реализовывать НЕ нужно.

## 5. 🐛 Критические баги (код-ревью 2026-06-16)

> **Ждёт:** доказательства прибыльности paper trade. Фиксить и улучшать код, пока
> стратегия не доказала edge, — пустая трата времени.

### 5.1 `paper_repo.py` — отравление пула соединений
Все 17+ функций: `get_conn() → try → commit → finally → put_conn()`. При ошибке
между `get_conn()` и `commit()` соединение возвращается в пул **в состоянии aborted
transaction** — psycopg2 отвергнет все последующие запросы на нём.
**Фикс:** добавить `except: conn.rollback(); raise`.

### 5.2 `paper_loop.py` — race condition на circuit breaker
`record_trade_result()` вызывает `get_state()` → модифицирует → `update_state()` в **двух
транзакциях**. Если две позиции закрываются в одной итерации — одно обновление теряется,
CB может не сработать при 5 подряд убытках.
**Фикс:** атомарный `UPDATE ... SET consec_losses = consec_losses + 1 RETURNING`.

### 5.3 `reconcile.py` — reconciled-позиции записывают PnL = 0
Позиции, закрытые вне бота (мануал, ликвидация, экспирация), получают `pnl_usd=0.0`.
Искажает статистику (`position_stats`) и **обходит daily loss limit** — убыток от ликвидации
не учитывается.
**Фикс:** рассчитывать PnL на основе `entry_price` + mark/intrinsic.

### 5.4 `broker.py` — нет exception handling вокруг вызовов биржи
`sell_to_open()` / `buy_to_close()` могут бросить сетевые/API исключения. Необработанное
исключение mid-retry оставляет систему в неконсистентном состоянии.
**Фикс:** try/except с логом, пометка позиции для reconcile.

### 5.5 `paper_loop.py` — нет лимита на количество позиций
Есть margin cap 80%, но нет жёсткого `MAX_OPEN_POSITIONS`. В волатильном рынке может
открыть десятки позиций. Пересекается с п.2 (лимит концентрации).
**Фикс:** `MAX_OPEN_POSITIONS` в `execution_config.py` (рекомендация: 5–10).

### 5.6 `paper_loop.py` — деление на ноль `entry_credit_pct`
Строка 341: `entry_credit_pct = ... / spot * 100` — paper-ветка не проверяет `spot > 0`
(live-ветка проверяет). При `spot == 0` → `ZeroDivisionError`.

### 5.7 `paper_repo.py` — `open_position` возвращает `0` при ошибке
`return int(row[0]) if row else 0` — `0` выглядит как валидный ID. Caller может работать
с несуществующей позицией. **Фикс:** возвращать `None`.

## 8. 📈 Улучшение профита: Динамический Sizing по ADX Score

**Статус:** проанализировано, прогнан sweep-бэктест на 365 днях (2026-06-16). Ждёт реализации.

### Описание
Текущая стратегия V2 Hybrid использует ADX/regime как бинарный фильтр (range/transition = торгуем, trend = не торгуем).
Разработан градиентный **ADX Readiness Score (0-10)** (учитывает значение ADX, угол наклона за 6ч и спред +DI/-DI).

По результатам бэктестов абсолютным победителем стал подход **"Baseline + Sizing"**:
1. Полностью сохраняем текущий бинарный фильтр (сохраняет 582 сделки/год, плавную кривую и всего 2 убыточных месяца).
2. Используем ADX Score для изменения размера позиции:
   - `Score 0-7`: обычный объем (1.0x).
   - `Score 8-10` (идеальные условия): увеличенный объем (1.5x).

**Результаты бэктеста:**
- **Без сайзинга (текущая):** Прибыль +$5,547, Средняя на сделку +$9.53.
- **С сайзингом:** Прибыль **+$7,181**, Средняя на сделку **+$12.34**.

### Что нужно сделать
Внедрить логику множителя в `live_sizing.py`:
- `adx_score` уже рассчитывается и прикрепляется к словарю `extra` сигнала (в `strategy_registry.py`).
- В функции `realistic_size_lots` (или аналоге) проверять `signal.get('adx_score', 0)`.
- Если `score >= 8`, применять множитель `1.5` к итоговому размеру позиции (не превышая глобальные лимиты).
