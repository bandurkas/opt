# AGENTS.md — правила разработки для любого агента в этом репо

> Это не снепшот состояния (то — `START_HERE.md` → `SESSION_STATE.md` → `ROADMAP.md`).
> Это **durable** правила: они не должны устаревать от сессии к сессии. Если правило здесь
> противоречит тому, что ты видишь в коде — код важнее, но сначала проверь, не сломал ли кто-то
> правило, и сообщи пользователю.

Три бота продают опционную премию на Bybit реальными/paper деньгами:
**Boba1** (BTC straddle), **Grogu1** (ETH straddle), **Sniper1** (ETH signal hybrid).
Все три — общий FastAPI+Postgres backend на VPS3 (`187.127.114.34`, репо `/root/opt-app`),
управляются дашбордом Mission Control. Деньги настоящие или становятся настоящими — каждое
правило ниже выросло из реального инцидента, не из абстрактной осторожности.

## 1. Edge (входы/выходы стратегии) — ничего без train/holdout

Любое изменение, которое трогает: пороги входа, TP/SL, sizing-формулу, strike-выбор, фильтры
регайма — **не катится без бэктеста на полной истории + holdout (OOS)**, и holdout проверяется
ПОСЛЕ того как порог/правило зафиксировано на train, а не подбирается по holdout. Несколько
рычагов уже были отклонены именно потому что train выглядел хорошо, а holdout — нет
(strangle, ADX-sizing, entry-frequency, perp-direction overlays — все отвергнуты тестами, не
интуицией). Если нет свежего train/holdout прогона под рукой — не трогай эти параметры, даже
если "кажется очевидным".

## 2. Workflow для любого изменения: архитектура → код → ревью → тест → ревью → деплой

Для live-money-adjacent изменений (Mission Control, ключи, sizing, execution.py) — минимум
2-3 раунда ревью перед деплоем. Это не бюрократия для галочки: `execution.py`'s missing
`orderLinkId` — реальный баг, который тихо проваливал КАЖДЫЙ ордер (Bybit отвечал ErrCode 10001),
и об этом узнали только потому что протестировали реальным (отменяемым) тест-ордером, а не
просто прочитали код и решили что он выглядит правильно.

## 3. Никогда не финализируй фикс как hot-patch в живом контейнере

`docker cp` в работающий контейнер — допустимая временная мера, чтобы не ждать ребилда, но
**конечное состояние — всегда в образе**, через `git commit` → `docker compose build` →
`docker compose up -d --no-build --force-recreate`. Если ты hot-patch'нул контейнер — это
TODO, не done, и должно быть явно в todo-листе/памяти, иначе следующий ребилд молча откатит фикс.

## 4. Деплой никогда не идёт мимо guard-скрипта при открытых позициях

`scripts/deploy_on_vps.sh` проверяет открытые позиции перед `docker compose down`/restart и
откажется выполняться, если они есть (обходится только явным `DEPLOY_FORCE=1`, и это осознанное
решение, не привычка). Причина: рестарт контейнера с открытыми позициями без сверки состояния —
прямой путь потерять отслеживание реальной позиции на бирже. Не зови `docker compose down`
вручную в обход скрипта на проде.

## 5. Каждый бот = свой Bybit-аккаунт. Никогда не делить капитал/ключи между ботами

Архитектура на это рассчитана (`db/accounts_repo.py`, `MC_ACCOUNT_NAME` per service в
`docker-compose.yml`). `reconcile.py` хранит позиции по coin-ключу — это известная "мина" при
смешивании капитала нескольких ботов на одном coin/аккаунте; решение — НЕ патчить `reconcile.py`,
а держать ботов на раздельных аккаунтах (так и сделано: Sniper1 и будущий ETH straddle — отдельные
ключи). Не пытайся "упростить" объединением счетов.

## 6. Sizing: округление в 0 — реальный, не гипотетический баг-класс

Любая sizing-формула, которая делит equity/margin на price и округляет qty до шага лота
(`qtyStep`/`minOrderQty`), может округлиться в 0 при тонком капитале или дорогом контракте —
найдено и исправлено живьём в `paper_strategy.py::realistic_size_lots` (margin-starvation bug,
2026-06-23/24). При написании любой sizing-логики — явно проверяй пол лота, не доверяй "обычно
этого не случится".

## 7. SL — на марже/equity, не на % премии

%-of-premium SL ломается на gap-риске (премия может прыгнуть в разы на тонкой ликвидности раньше,
чем SL успеет сработать по проверяемой цене) — это вызвало реальный убыток -$24.58 на Grogu
(2026-06-23, см. `finding_grogu_sl_incident_deploy_gap`). Текущий паттерн — dollar-margin SL
(абсолютный $ trip относительно открытой маржи), это намеренный дизайн, не наследие — не
откатывай на %-SL без переоткрытия этого вопроса с пользователем.

## 8. Секреты — никогда в репо

Bybit API-ключи / пароли — только в `/root/opt-app/.env` на VPS3 (не в git) + зашифрованно в
Postgres (`exchange_credentials`, Fernet, ключ `CREDENTIALS_MASTER_KEY`). `HANDOFF.md` в этом
репо содержит исторический утёкший пароль VPS — он устарел и это файл-предупреждение, не
шаблон для новых доков.

## 9. Local / GitHub / VPS3 — держать в синке явно

VPS3 — рабочий клон того же репо (`git@github.com:bandurkas/opt.git`), не редактируется вручную
напрямую (правок прямо в файлах на VPS быть не должно — только `git pull` после push). Если
застаёшь VPS3 в состоянии, отличном от `origin/main` — это сигнал расследовать, не "просто
git reset --hard" сходу.

## 10. Сборка образов

Сейчас (с момента как VPS3 обновили до 2 CPU) — собирать нативно на VPS3, это быстро. Старое
правило "собирать на Mac, потому что VPS — 1 CPU и `docker compose build` стопорит SSH на
40+ минут" устарело для opt-app (VPS3, 2 CPU), но **остаётся актуальным для других проектов на
других VPS** (например opt21 на VPS2, который всё ещё 1 CPU) — не путай инстансы.

## 11. Где смотреть состояние, а не правила

`START_HERE.md` → `SESSION_STATE.md` → `ROADMAP.md` — текущий снимок (что сделано, что
открыто, доступы, команды). Эти файлы обновляются каждую сессию и могут отставать на день-два;
если что-то здесь противоречит свежему `git log`/живому состоянию контейнеров — доверяй живому
состоянию и обнови снепшот.
